## 1. НАЗНАЧЕНИЕ ДОКУМЕНТА

Настоящий план определяет порядок действий при критическом сбое IT-инфраструктуры компании, включая атаки вирусов-шифровальщиков, аппаратные отказы, пожар или затопление серверного помещения.

**Область применения:** все IT-системы организации, перечисленные в Разделе 3.

**Цель:** обеспечить возобновление ключевых бизнес-процессов в установленные сроки с минимальными потерями данных.

---

## 2. ЦЕЛЕВЫЕ ПОКАЗАТЕЛИ ВОССТАНОВЛЕНИЯ

| Показатель | Определение | Целевое значение |
|---|---|---|
| **RPO** (Recovery Point Objective) | Максимально допустимые потери данных по времени | `[например: 24 часа]` |
| **RTO** (Recovery Time Objective) | Максимально допустимое время восстановления | `[например: 3 рабочих дня]` |
| **MTTR** | Среднее время восстановления одной системы | `[например: 4 часа]` |

> **Как выбрать значения:** RPO определяется тем, какой объём работы сотрудники готовы повторить вручную. Если вы работаете с 1С каждый день — RPO не должен превышать 24 часа. RTO определяется тем, сколько дней простоя выдержит бизнес финансово.

## 7. ПОРЯДОК ДЕЙСТВИЙ ПРИ КРИТИЧЕСКОМ ИНЦИДЕНТЕ

### Фаза 1. Обнаружение и оповещение (0–30 минут)

- [ ] Зафиксировать время и характер инцидента
- [ ] Позвонить IT-ответственному: `[номер]`
- [ ] IT-ответственный уведомляет директора: `[номер]`
- [ ] Оценить масштаб: сколько систем затронуто

### Фаза 2. Сдерживание (30 минут – 2 часа)

- [ ] **При атаке шифровальщика:** физически отключить заражённые устройства от сети (кабель из розетки)
- [ ] **При аппаратном сбое:** зафиксировать симптомы, не перезагружать без необходимости
- [ ] **При пожаре / затоплении:** эвакуация людей — приоритет, оборудование вторично
- [ ] Убедиться, что носители с резервными копиями в безопасности
- [ ] Уведомить сотрудников: не включать компьютеры, не подключаться к сети

### Фаза 3. Оценка (2–4 часа)

- [ ] Инвентаризировать повреждения (заполнить таблицу из Раздела 3)
- [ ] Проверить наличие и состояние резервных копий
- [ ] Определить приоритетный порядок восстановления
- [ ] Оценить необходимость привлечения внешних специалистов
- [ ] Составить и согласовать с директором план восстановления

### Фаза 4. Восстановление

**Приоритетный порядок для малой компании:**

```
1. Интернет и сеть (маршрутизатор, коммутатор)
2. Сервер / файловый сервер
3. 1С и базы данных
4. Рабочие места — начиная с бухгалтерии
```

- [ ] Восстановление сети и маршрутизатора
- [ ] Восстановление сервера из резервной копии
- [ ] Проверка целостности данных (особенно 1С — тестовый вход)
- [ ] Последовательное восстановление рабочих мест
- [ ] Проверка корпоративного ПО на каждом рабочем месте

### Фаза 5. Верификация и возврат к работе

- [ ] Тестовый вход каждого сотрудника в свои системы
- [ ] Проверка актуальности данных (сравнить с последними известными документами)
- [ ] Смена всех паролей
- [ ] Официальное объявление о возобновлении работы
- [ ] Составление акта об инциденте

## 8. СЦЕНАРИИ И ЧАСТНЫЕ ИНСТРУКЦИИ

### Сценарий А. Атака вируса-шифровальщика

**Симптомы:** файлы изменили расширение, появился файл README с требованием выкупа, антивирус сообщает об угрозе.

**Первые 15 минут:**
1. Немедленно отключить сетевой кабель от всех компьютеров, на которых заметили симптомы
2. НЕ выключать компьютеры кнопкой питания
3. НЕ пытаться «почистить» антивирусом
4. НЕ платить выкуп без консультации со специалистом
5. Позвонить IT-ответственному

**Если бэкапы в порядке:** восстановление по Фазе 4 выше.  
**Если бэкапы недоступны или зашифрованы:** связаться со специалистами: **sos@nullday.ru**

### Сценарий Б. Аппаратный отказ сервера

**Симптомы:** сервер не загружается, синий экран, физические повреждения.

1. Зафиксировать симптомы (фото, скриншот экрана ошибки)
2. Проверить кабели питания и сигнальные кабели
3. Попытаться загрузиться с диагностического носителя
4. Если сервер не восстанавливается — разворачивать на резервном оборудовании из бэкапа
5. Связаться с поставщиком оборудования для замены или ремонта

### Сценарий В. Отказ интернет-канала

1. Проверить, не отключилось ли питание у роутера и ONT-терминала провайдера
2. Позвонить провайдеру: `[номер техподдержки]`
3. Если провайдер не отвечает — активировать резервный канал (если есть): `[описание]`
4. При длительном простое — рассмотреть мобильный интернет как временное решение

### Сценарий Г. Пожар или затопление

1. **Приоритет — эвакуация людей**
2. Вызвать экстренные службы: 112
3. По возможности — отключить питание серверной
4. Зафиксировать ущерб для страховой
5. Начать восстановление только после получения доступа к помещению и разрешения компетентных органов
6. Восстановление — на новом оборудовании из внеофисных резервных копий
