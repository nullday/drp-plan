## 6. КЛАССИФИКАЦИЯ И ОБНАРУЖЕНИЕ ИНЦИДЕНТОВ

### 6.1 Уровни инцидентов

| Уровень | Критерии | Примеры | Время реакции IT |
|---|---|---|---|
| 🔴 P1 — Катастрофа | Полная остановка бизнеса | Ransomware, пожар серверной, отказ всех серверов | 15 минут |
| 🟠 P2 — Критический | Недоступна ключевая система | Падение DC, отказ сервера 1С | 30 минут |
| 🟡 P3 — Значительный | Деградация сервиса | Замедление сети, недоступность части папок | 2 часа |
| 🟢 P4 — Незначительный | Единичная проблема | Один компьютер не работает | 1 рабочий день |

### 6.2 Признаки атаки шифровальщика

Немедленно объявить инцидент P1 при обнаружении:
- Файлы изменили расширение (`.locked`, `.encrypted`, `.STOP`, и т.п.)
- На рабочем столе или в папках появился файл с инструкцией по выкупу
- Антивирус выдаёт множественные предупреждения
- Резкое замедление всех серверов без видимой причины
- Сотрудники массово сообщают о проблемах с файлами

## 7. ПОШАГОВЫЕ ПРОЦЕДУРЫ ВОССТАНОВЛЕНИЯ

### 7.1 Процедура P1: Полное восстановление (атака / катастрофа)

#### Час 0–1: Сдерживание

- [ ] Немедленная изоляция: отключить кабели от заражённых устройств
- [ ] Оповестить всю команду восстановления
- [ ] Запретить сотрудникам включать компьютеры
- [ ] Изолировать NAS с бэкапами (физически отключить от сети)
- [ ] Идентифицировать штамм (если ransomware): [id-ransomware.malwarehunterteam.com](https://id-ransomware.malwarehunterteam.com)
- [ ] Зафиксировать всё: фото экранов, журналы, временные метки

#### Час 1–3: Оценка

- [ ] Составить полный список заражённых / отказавших систем
- [ ] Проверить состояние и актуальность резервных копий
- [ ] Определить точку входа (вектор атаки)
- [ ] Согласовать план и очерёдность с директором
- [ ] Принять решение о привлечении внешних специалистов

#### Час 3+: Восстановление по очерёдности

**Этап 1. Сеть (4–6 часов)**
- [ ] Полный сброс MikroTik (`/system reset-configuration`)
- [ ] Восстановление конфигурации из последнего бэкапа
- [ ] Если бэкапа нет — настройка с нуля по документации
- [ ] Проверка: интернет работает, маршрутизация внутри сети — работает
- [ ] Блокировка внешнего RDP, смена паролей оборудования

**Этап 2. Active Directory (4–8 часов)**
- [ ] Чистая установка Windows Server
- [ ] Восстановление AD из System State бэкапа
- [ ] Двукратный сброс пароля krbtgt
- [ ] Сброс паролей всех пользователей домена
- [ ] Проверка: `dcdiag /v` без ошибок
- [ ] Проверка членства в группе Domain Admins — только доверенные

**Этап 3. Файловый сервер (2–4 часа + время копирования)**
- [ ] Чистая установка ОС или разворачивание из бэкапа
- [ ] Восстановление данных из резервной копии
- [ ] Настройка SMB-шар и прав доступа
- [ ] Проверка: сотрудники видят и открывают папки

**Этап 4. Сервер 1С (3–6 часов)**
- [ ] Установка SQL Server
- [ ] Восстановление баз из `.bak` бэкапа
- [ ] Установка и настройка сервера 1С
- [ ] Тестирование с главным бухгалтером
- [ ] Проверка: можно зайти во все ИБ, данные актуальны

**Этап 5. Рабочие места (1–2 часа на место)**
- [ ] Приоритет: бухгалтерия → руководство → остальные
- [ ] Чистая установка ОС
- [ ] Ввод в домен
- [ ] Установка корпоративного ПО
- [ ] Подключение сетевых папок через GPO

**Этап 6. Закрытие уязвимости**
- [ ] Установить точку входа и устранить её
- [ ] Обновить все системы и ПО
- [ ] Настроить EDR / антивирус на всех узлах
- [ ] Провести инструктаж сотрудников

### 7.2 Процедура P2: Восстановление отдельной системы

**Отказ сервера 1С:**
1. Проверить службы SQL Server и сервера 1С (диспетчер задач, Event Log)
2. Попытаться перезапустить службы
3. Проверить место на диске (SQL-лог может переполниться)
4. Если не помогает — восстановить из последнего бэкапа БД

**Отказ контроллера домена:**
1. Проверить физическое состояние сервера
2. Попытаться загрузиться в безопасном режиме
3. Если недоступен — поднять резервный DC (если есть) или восстановить из System State

**Отказ интернет-канала:**
1. Проверить статус ONT/роутера провайдера
2. Позвонить провайдеру: `[номер]`
3. Переключить на резервный канал: `[описание процедуры]`
